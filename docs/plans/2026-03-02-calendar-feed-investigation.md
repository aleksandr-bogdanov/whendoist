# iCal (.ics) Calendar Subscription Feed — Investigation

## Findings

### 1. Existing Calendar Code

**No iCal/ics handling exists anywhere in the codebase.** The calendar-related code is:

- **`app/services/gcal.py`** — `GoogleCalendarClient` that talks to Google Calendar API v3 via `httpx`. Read-only methods (`list_calendars`, `get_events`) and write methods (`create_event`, `update_event`, `delete_event`). No iCal generation.
- **`app/services/calendar_cache.py`** — In-memory cache for Google Calendar *read* events (5-minute TTL, keyed by user + date range + calendar IDs). Purely for caching GCal API responses.
- **`app/routers/gcal_sync.py`** — Router for enabling/disabling/triggering one-way Whendoist→GCal sync.
- **`app/services/gcal_sync.py`** — Service that pushes tasks to Google Calendar as events.

There are no `events.py` or `calendars.py` routers — those don't exist.

### 2. GCal Sync — Event Construction

The `build_event_data()` function in `gcal.py:497` constructs Google Calendar event payloads:

```python
def build_event_data(title, description, scheduled_date, scheduled_time,
                     duration_minutes, is_completed, user_timezone) -> dict:
```

- **Timed tasks**: `scheduled_date` + `scheduled_time` → `dateTime` start/end (using `duration_minutes` or default 30min)
- **All-day tasks**: `scheduled_date` only → `date` start (next day end, per GCal convention)
- **Completed**: prefixes title with `✓`

This logic would be **directly reusable** for .ics generation — the same field mapping applies. The `compute_sync_hash()` function could also serve as a cache invalidation key.

### 3. Task Data Shape for Calendar

**Non-recurring tasks** (`Task` model, `app/models.py:291`):
| Field | Type | Notes |
|---|---|---|
| `scheduled_date` | `Date` (nullable) | The calendar date |
| `scheduled_time` | `Time` (nullable) | Time of day (null = all-day) |
| `duration_minutes` | `Integer` (nullable) | Duration (null = default 30min for timed, or all-day) |
| `title` | `Text` | May be encrypted ciphertext |
| `description` | `Text` (nullable) | May be encrypted ciphertext |
| `status` | `String` | pending / completed / archived |

**Recurring tasks** use `TaskInstance` (`app/models.py:368`):
| Field | Type | Notes |
|---|---|---|
| `instance_date` | `Date` | The concrete occurrence date |
| `scheduled_datetime` | `DateTime` (nullable) | UTC datetime if parent has `scheduled_time` |
| `status` | `String` | pending / completed / skipped |

Instances **are** already materialized with concrete dates (60-day rolling window, generated by `RecurrenceService.materialize_instances()`). The parent task holds `scheduled_time` and `duration_minutes`. This means the feed can query instances directly without re-computing recurrence rules.

**GCal sync's bulk query** (`gcal_sync.py:487-510`) is the exact pattern we'd reuse:
- Non-recurring: `Task` where `scheduled_date IS NOT NULL` or `completed_at IS NOT NULL`, status ≠ archived
- Recurring instances: `TaskInstance` joined to `Task`, status ≠ skipped, parent ≠ archived

### 4. Auth for Unauthenticated Endpoints

**Current auth**: All API routes use `Depends(require_user)` which reads `user_id` from a signed session cookie (`itsdangerous` + Starlette `SessionMiddleware`). There is **no existing pattern** for token-based URL auth.

**What calendar apps need**: A stable URL like `https://whendoist.com/api/v1/calendar-feed/abc123.ics` that works with a plain GET request — no cookies, no headers, no OAuth.

**Proposed approach**: Generate a per-user random token (e.g., `secrets.token_urlsafe(32)`) stored in a new column on `UserPreferences` (or a dedicated model). The token goes in the URL path. The endpoint validates the token, looks up the user, and serves the feed. The token can be regenerated (invalidating old URLs) if the user wants to revoke access.

**Security considerations**:
- Token must be cryptographically random and long enough to prevent brute-force (32 bytes = 256 bits)
- HTTPS ensures the URL isn't sniffed in transit
- No CSRF concern since it's read-only
- Rate limiting is still advisable to prevent enumeration
- Token rotation should be a user-facing action ("Regenerate feed URL")

### 5. Encryption Complication

**GCal sync already handles this explicitly:**

1. **Title masking**: When `encryption_enabled`, titles become `"Encrypted task"` (constant `GCAL_SYNC_ENCRYPTED_PLACEHOLDER`)
2. **Enable blocking**: The `/gcal-sync/enable` endpoint **rejects** enabling sync when encryption is on: `"Cannot enable sync when E2E encryption is enabled. Task titles would be synced as encrypted text."`

**For the .ics feed, three options:**

| Option | Behavior | UX |
|---|---|---|
| A. Block entirely | Don't generate feed when encryption enabled | Clean but limits feature availability |
| B. Mask titles | Use placeholder like `"Encrypted task"` or `"Busy"` | Calendar shows time blocks without details |
| C. Allow ciphertext | Serve encrypted titles as-is | Useless to the user, terrible UX |

**Recommendation: Option B** (mask titles). Unlike GCal sync which is an opt-in push, the feed URL is something the user consciously subscribes to. Showing "Busy" time blocks is actually useful — the user sees when they're occupied without leaking task details. This is exactly what Google Calendar's "free/busy" sharing does.

### 6. Python iCal Libraries

| Library | PyPI | GitHub Stars | Last Release | Python | Notes |
|---|---|---|---|---|---|
| **icalendar** | `icalendar` | ~1.7k | Active (2025) | 3.8+ | Classic, most downloaded, simple API for generation |
| **ics** | `ics` | ~500 | Active (2025) | 3.8+ | More pythonic, arrow-based, heavier |
| **ical** | `ical` | ~100 | Feb 2026 (v13) | 3.11+ | Newest, excellent for parsing, overkill for generation |
| **vobject** | `vobject` | ~200 | Forked, maintained | 3.8+ | Full vCard+iCal, heavier than needed |

**None are in the project's current `pyproject.toml` dependencies.**

**Recommendation: `icalendar`**. For *generating* (not parsing) .ics files, it's the simplest and most widely used. Minimal API:

```python
from icalendar import Calendar, Event

cal = Calendar()
cal.add("prodid", "-//Whendoist//EN")
cal.add("version", "2.0")

event = Event()
event.add("summary", "My Task")
event.add("dtstart", date(2026, 3, 5))
event.add("dtend", date(2026, 3, 6))
cal.add_component(event)

ics_bytes = cal.to_ical()  # bytes ready to serve
```

Adding `icalendar>=6.0.0` to dependencies is lightweight (~50KB, pure Python, depends only on `python-dateutil` which is already a dependency).

### 7. Caching and Polling

**Calendar app behavior**: Apple Calendar polls every 15 min (configurable), Google Calendar every 12-24 hours, Outlook every 30 min. This means the endpoint gets hit ~50-100 times/day per subscriber.

**Load analysis for generation-per-request**:
- Query: 2 SQL queries (tasks + instances) — same as GCal bulk_sync does
- Serialization: `icalendar` to_ical() is fast (pure string building)
- For a user with 200 tasks + 500 instances: <50ms total
- At 100 requests/day: negligible load

**Caching strategy**:

| Strategy | Complexity | Freshness |
|---|---|---|
| Generate per request | Low | Real-time |
| Cache with `data_version` key | Medium | Near-real-time (invalidated on any task mutation) |
| Pre-generate on task mutation | High | Instant but adds write-path complexity |

**Recommendation: Generate per request, with optional ETag/304 optimization.** The `User.data_version` field (bumped on every mutation) is a perfect ETag — if the calendar app sends `If-None-Match` with the last version, return `304 Not Modified`. This avoids regeneration on most polls.

### 8. What Should Be in the Feed

**Include (matches GCal sync scope):**
- Scheduled tasks (pending and completed) with `scheduled_date`
- Completed tasks with `completed_at` date (Todoist imports that lack `scheduled_date`)
- Recurring task instances (pending and completed, **not** skipped)
- All-day events for date-only tasks
- Timed events with duration for tasks with `scheduled_time`

**Exclude:**
- Archived tasks
- Unscheduled tasks (no date = no calendar event)
- Skipped recurring instances
- Subtasks (debatable — could include if they have their own scheduled_date)

**Event metadata to include:**
- `SUMMARY`: Task title (or "Busy" if encrypted)
- `DESCRIPTION`: Task description
- `DTSTART` / `DTEND`: From scheduled_date + scheduled_time + duration
- `STATUS`: COMPLETED / NEEDS-ACTION (maps to iCal VTODO, though VEVENT is more universal for calendar display)
- `UID`: Stable unique ID (e.g., `task-{id}@whendoist.com` or `instance-{id}@whendoist.com`)
- `LAST-MODIFIED`: Task's `updated_at`

**Use VEVENT (not VTODO)**: Most calendar apps display VEVENTs prominently but hide/ignore VTODOs. Since the goal is to see tasks *on the calendar*, VEVENT is the right choice.

## Options

### Option A: Simple Read-Only Feed
- New endpoint: `GET /api/v1/calendar-feed/{token}.ics`
- Generate .ics on each request
- ETag via `data_version` for 304 responses
- Mask titles when encryption enabled
- ~200 lines of new code (endpoint + service)
- 1 new dependency (`icalendar`)
- 1 new DB column (`feed_token` on UserPreferences)
- Frontend: "Copy feed URL" button in settings

### Option B: Cached Feed with Background Generation
- Same endpoint, but pre-generates .ics on task mutations
- Store serialized .ics in DB or Redis
- More complex: mutation hooks, storage, invalidation
- Only worthwhile at scale (100+ active users polling frequently)

### Option C: Feed with Selective Filtering
- Like Option A, but with query parameters for filtering
- `?domains=1,2,3` — only tasks from specific domains
- `?status=pending` — only pending tasks
- `?range=30d` — only next 30 days
- More complex but more useful for power users
- Calendar apps would need separate subscriptions per filter

## Open Questions

1. **Subtasks**: Should subtasks with their own `scheduled_date` appear as separate events? Currently GCal sync includes them (they're just tasks with `parent_id` set).

2. **Completed task retention**: Should completed tasks stay in the feed forever, or drop off after N days? GCal sync keeps them, but a feed that grows unboundedly could slow down calendar apps. Suggestion: include completed tasks from the last 30 days only.

3. **Domain colors**: iCal supports `COLOR` property (RFC 7986). Should tasks inherit their domain's color? Most calendar apps ignore this, but Apple Calendar respects it.

4. **VALARM (reminders)**: Should the feed include reminders? This could be useful but risks notification spam if the user already has Whendoist notifications.

5. **Feed scope**: One feed per user (all tasks), or per-domain feeds? Per-domain is more flexible but adds complexity and multiple URLs to manage.

6. **Token storage**: New column on `UserPreferences` vs. a dedicated `CalendarFeedToken` model? The former is simpler; the latter allows multiple feeds (e.g., per-domain).

7. **Migration**: Needs a new migration for the feed token column. Should the token be auto-generated on first access, or explicitly created via a "Generate feed URL" action?

---

## Recommendation

**Start with Option A (Simple Read-Only Feed)** — it covers 90% of the use case with minimal complexity.

### Implementation sketch:

1. **Add `icalendar>=6.0.0`** to `pyproject.toml` dependencies
2. **Add `feed_token`** column to `UserPreferences` (nullable String, indexed)
3. **New service** `app/services/calendar_feed.py`:
   - Query tasks + instances (reuse GCal sync's query pattern)
   - Build `icalendar.Calendar` with VEVENT components
   - Handle encryption masking (same pattern as GCal sync)
4. **New endpoint** (outside the `require_user` dependency chain):
   - `GET /api/v1/calendar-feed/{token}.ics`
   - Look up user by token, return 404 if invalid
   - Set `Content-Type: text/calendar; charset=utf-8`
   - Set `ETag` from user's `data_version`, support `If-None-Match` → 304
   - Rate limit by IP
5. **Frontend**: "Calendar Feed" section in Settings with:
   - "Generate Feed URL" button (creates token if none exists)
   - Copy-to-clipboard URL display
   - "Regenerate URL" to revoke old URL
   - Warning when encryption is enabled ("titles will show as 'Busy'")

### Effort estimate scope:
- Backend: ~200 lines (service + endpoint + migration)
- Frontend: ~100 lines (settings UI)
- 1 new dependency, 1 DB migration
- No changes to existing code paths
